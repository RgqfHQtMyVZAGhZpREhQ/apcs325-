Q-1-10 · 最多得分的皇后
用全域變數收集答案，不同路徑互相污染
高風險

最初的寫法用一個全域變數記錄目前看過的最大值，但 rec(k) 在整個遞迴過程裡會被不同的父呼叫重複呼叫很多次。某一條路徑更新的全域值，會被另一條完全無關的路徑誤用，導致算出不存在的「混合答案」。

int big {-140000};   // 全域變數

int maxq(... k ...) {
    ...
    for (合法的 i) {
        p[k] = i;
        total = maxq(score, k+1, p) + score[k][i];
        max(total, big);          // 連賦值都忘了
    }
    return big;                   // 回傳的是全域狀態,不是這層自己的答案
}
int maxq(... k ...) {
    int best = empt;   // 區域變數,只屬於這一層呼叫
    for (合法的 i) {
        p[k] = i;
        total = maxq(score, k+1, p) + score[k][i];
        best = max(best, total);
    }
    return best;        // 回傳「這一層自己算出的局部最佳解」
}
原則：遞迴函式要回傳「這一次呼叫自己算出的局部答案」，不要依賴全域狀態收集結果——不同分支會互相污染。
Q-1-10 · 最多得分的皇后
迴圈完全沒執行時，「跳過」這個選項憑空消失
高風險 · 曾經 AC 但答案是錯的

當某一列所有位置都被攻擊，for 迴圈一次都不會執行，best 就停在初始值 0，從來沒機會跟 empt（跳過這列的分數）比較。這個版本在 APCS 上 AC 了——純粹是因為判題測資沒覆蓋到這個邊界，不代表邏輯是對的。

int best {};                 // 初始值是 0
empt = maxq(score, k+1, p);

for (合法的 i) {
    total = maxq(...) + score[k][i];
    best = max(max(total, best), empt);   // empt 只在迴圈裡才被比較
}
return best;   // 若迴圈沒跑過,empt 永遠沒被用到
empt = maxq(score, k+1, p);
int best = empt;             // 起始值就是 empt,跳過這列「先站上起跑線」

for (合法的 i) {
    total = maxq(...) + score[k][i];
    best = max(max(total, best), empt);
}
return best;   // 即使迴圈一次都不執行,也已經正確包含跳過的分數
原則：每一個合法選項都要保證會被比較到，不能依賴「迴圈一定會執行」這種沒寫出來的隱藏假設。
Q-1-11 · 刪除矩形邊界
用貪心選「當下最便宜」的邊，而不是窮舉比較
高風險 · 演算法層級的錯誤

第一版邏輯是每一層都選成本最小的那條邊先刪掉。但反例很容易構造：上下兩條邊界現在都是「免費」(成本0)，刪掉其中一個之後，剩下那條邊界的內容會完全改變，新的成本可能暴增。當下最便宜，不代表全局最便宜。

maaax = min(max(t,b), max(l,r));   // 貪心:只挑一條邊
if (maaax == t)
    total = maaax + slove(top+1, bottom, left, right);
else if (maaax == b)
    total = maaax + slove(top, bottom-1, left, right);
... // 其餘三條邊只能擇一往下走,其他分支被永遠捨棄
// 四條邊全部都試,各自遞迴,最後取總成本最小的那條路
total = min({
    t + slove(top+1, bottom, left, right),
    b + slove(top, bottom-1, left, right),
    l + slove(top, bottom, left+1, right),
    r + slove(top, bottom, left, right-1)
});
原則：會牽動未來狀態的決策問題，局部最優不保證全局最優。要窮舉所有分支、各自遞迴，再從結果裡取最小（或最大）。
Q-1-11 · 刪除矩形邊界
多變數宣告初始化陷阱：{} 只作用在最後一個變數
隱藏陷阱 · 小測資會矇混過關

int cz, co {}; 這樣寫，只有 co 會被初始化成 0，cz 完全沒有初始化，內容是未定義的垃圾值。小矩陣時剛好沒事，矩陣一變大、遞迴變深，垃圾值就會讓計數整個跑掉，跑出離譜的負數。用 sanitizer 重現過這個 bug，行為完全符合「未初始化變數」的特徵。

int cz, co {};     // 只有 co 被初始化,cz 是垃圾值
int czs, cos{};    // 只有 cos 被初始化,czs 是垃圾值
int cz{}, co{};      // 每個變數都自己寫一次 {}
int czs{}, cos{};
原則：C++ 裡 int a, b {}; 的 {} 只初始化緊接著的那一個變數。要全部初始化，每個變數都要自己加。這種 bug 編譯器不會警告，而且結果隨記憶體狀態而變，最難抓。
03

可以帶走的 5 個原則

不只是這幾題的解法，是寫遞迴 / 窮舉題目時可以重複套用的檢查清單。
01
回傳局部解，不要靠全域變數
每次遞迴呼叫只對自己負責，算出自己的答案往上傳，不要用共享的全域狀態收集結果。
02
每個選項都要保證被比較到
不要依賴「迴圈一定會執行」這種沒明寫的假設——零次執行的情況要單獨確認。
03
局部最優 ≠ 全局最優
會牽動未來狀態的決策，貪心當下最好的選擇不保證整體最好，要窮舉所有分支取最佳。
04
多變數宣告，初始化要逐一寫
int a, b{}; 只初始化最後一個，要全部初始化就每個變數都自己加 {}。
05
AC 不等於正確
判題測資不一定覆蓋所有邊界情況。自己多想幾個邊界案例（全 0、全 1、剛好擋死等）比單純拿到 AC 更可靠。
所有程式碼皆以 g++ -O2 搭配 AddressSanitizer / UndefinedBehaviorSanitizer 重新編譯驗證，並補測邊界案例（單列/單行、完全擋死、大矩陣等）。AP325 第一章 · 遞迴。
